Chapter 01

Project Overview

What is Aqua Sort and how does it work at a high level?

Aqua Sort is a full-stack web application — meaning it has two parts that work together: a frontend (what the user sees in the browser) and a backend (a server running Python that handles the game logic).

🧠 Key Concept — Full Stack

Frontend = HTML, CSS, JavaScript running in your browser. It draws the game and reacts to clicks. Backend = Python server running somewhere else. It validates moves, runs the AI, stores data. They talk to each other using a REST API — basically sending JSON messages back and forth over the internet.

API Calls with fetch()

JavaScript uses the built-in fetch() function to talk to the Flask backend:

JavaScriptapp.js
// All API calls go through this helper function
async function apiFetch(endpoint, method = "GET", body = null) {
  const opts = { method, headers: { "Content-Type": "application/json" } };
  if (body) opts.body = JSON.stringify(body);  // convert JS object to JSON text

  const res = await fetch(`${API_BASE}${endpoint}`, opts);
  return await res.json();  // parse JSON text back to JS object
}

// Example: making a move
const data = await apiFetch("/move", "POST", {
  session_id: State.sessionId,
  from_tube: 0,
  to_tube: 2
});
💡 What is async/await?

API calls take time (the request travels to the server and back). async/await pauses the function until the response arrives, instead of blocking the entire browser. It makes asynchronous code look like synchronous code.

Backend — Flask (Python)

The web server that handles all game logic, data, and API requests.

File: backend/app.py

📖 What is Flask?

Flask is a Python library that lets you create a web server in just a few lines. You define routes (URL paths like /api/move) and functions that run when someone visits those URLs. It listens for HTTP requests and sends back HTTP responses — usually JSON data.

How a Flask route works

Pythonapp.py
# The @app.route decorator tells Flask "when someone calls POST /api/move,
# run the make_move() function"
@app.route("/api/move", methods=["POST"])
def make_move():
    data = request.get_json()     # read JSON from the request body
    session_id = data["session_id"]
    src = data["from_tube"]
    dst = data["to_tube"]

    engine = sessions[session_id]["engine"]
    success, reason, moved = engine.pour(src, dst)

    return jsonify({               # send JSON response back
        "success": success,
        "reason": reason,
        "move_count": engine.move_count,
        "is_won": engine.is_won(),
    })

Voice Command Parser

Uses Python's re (regular expressions) module to extract meaning from spoken text:

Pythonapp.py
import re

def parse_voice_command(text):
    t = text.lower().strip()

    # Check for "undo" anywhere in the text
    if re.search(r"\bundo\b", t):
        return {"action": "undo"}

    # Match "pour red into tube 2" pattern
    match = re.search(r"pour\s+(\w+)\s+(?:into|to)\s+tube\s+(\w+)", t)
    if match:
        color = match.group(1)   # "red"
        tube  = match.group(2)   # "2"
        return {"action": "pour_color", "color": color, "to_tube": tube}

Backend — Game Engine

The pure Python logic that makes the game work.

File: backend/game_engine.py

The Tube Class

Represents a single tube. It stores liquid layers as a Python list (index 0 = bottom):

Pythongame_engine.py
class Tube:
    def __init__(self, capacity=4):
        self.capacity = capacity
        self.layers = []   # e.g. ["red", "blue", "red"] — bottom to top

    def top_color(self):
        return self.layers[-1] if self.layers else None

    def is_complete(self):
        # Full AND all layers same color
        return self.is_full() and len(set(self.layers)) == 1

    def top_count(self):
        # How many consecutive same-color layers at the top
        color = self.top_color()
        return sum(1 for l in reversed(self.layers) if l == color)

Undo Stack

Before every move, a complete snapshot of all tube states is saved. Undo just pops the last snapshot:

Pythongame_engine.py
def pour(self, src, dst):
    # Save snapshot BEFORE making the move
    self.undo_stack.append(self.snapshot())

    # Make the move...
    color = self.tubes[src].top_color()
    layers_to_move = min(s.top_count(), d.free_space())
    for _ in range(layers_to_move):
        self.tubes[src].layers.pop()      # remove from source
        self.tubes[dst].layers.append(color) # add to destination

def undo(self):
    if not self.undo_stack: return False
    snap = self.undo_stack.pop()   # get last snapshot
    self.restore(snap)              # restore tube states
    return True

Backend — AI Solver

How the computer automatically solves any puzzle.

File: backend/ai_solver.py

BFS — Breadth-First Search

📖 What is BFS?

Imagine exploring a maze. BFS tries every possible move from the start, then every move from those positions, then the next level, and so on — like ripples in water. It guarantees finding the shortest possible solution because it explores paths level by level (fewest moves first).

Pythonai_solver.py
from collections import deque

def _bfs(self, start):
    queue = deque([start])       # queue of states to explore
    visited = {start.key()}     # set of states already seen

    while queue:
        state = queue.popleft()  # take from front (oldest first)

        for src, dst in state.legal_moves():
            next_state = state.apply_move(src, dst)
            if next_state.key() in visited: continue
            if next_state.is_won():
                return next_state.path  # 🎉 found solution!
            visited.add(next_state.key())
            queue.append(next_state)

    return None  # no solution exists

A* Algorithm

📖 What is A*?

A* improves on BFS by using a heuristic — an estimate of how close a state is to the solution. Instead of exploring everything equally, A* prioritizes states that look more promising. It uses a priority queue (heap) ordered by cost + estimate. This makes it much faster for complex puzzles.

AlgorithmWhen UsedGuaranteeSpeed
BFSSmall puzzles (complexity ≤ 24)Optimal (fewest moves)Slower on large puzzles
A*Large puzzles (complexity > 24)Near-optimalMuch faster with heuristic

State Hashing

To avoid exploring the same game state twice, each state is converted to a unique string:

Pythonai_solver.py
def key(self) -> str:
    # Convert tube state to a unique string
    # e.g. "red,blue|green,red||blue,blue"
    return "|".join(",".join(t["layers"]) for t in self.tubes)

Backend — SQLite Database

How game data is stored permanently.

File: backend/models.py

📖 What is SQLite?

SQLite is a database that stores data in a single file (game.db). Unlike MySQL or PostgreSQL, it needs no separate server — Python's built-in sqlite3 module reads and writes to it directly. It's perfect for projects that don't need to handle thousands of simultaneous users.

Database Tables

SQLmodels.py — table definitions
-- Stores saved game states
CREATE TABLE game_saves (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id  TEXT UNIQUE NOT NULL,
    level       INTEGER NOT NULL,
    state_json  TEXT NOT NULL,     -- entire game state as JSON string
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- Stores completed session analytics
CREATE TABLE analytics_records (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id      TEXT NOT NULL,
    level           INTEGER NOT NULL,
    time_taken      REAL,
    moves           INTEGER,
    undo_count      INTEGER,
    failed_attempts INTEGER,
    won             INTEGER,       -- 0 or 1 (SQLite has no boolean)
    heatmap_json    TEXT           -- {tube_index: clicks} as JSON
);

How Save/Load Works

1

After every move

The entire GameEngine state (all tube layers) is serialized to JSON and saved to the game_saves table with the player's session_id.

2

When player clicks "Load"

The session_id is sent to GET /api/load. The server queries SQLite for that session_id, deserializes the JSON, and recreates the GameEngine object.

3

Game continues

The loaded state is sent back to the frontend as a normal API response. The frontend renders the tubes exactly as they were when saved.

How the Python Flask server runs 24/7 on the internet.

📖 What is Render?

Render is a cloud hosting platform. You give it your GitHub repo, tell it how to start your app, and it runs it on a server in a data center. Anyone in the world can then make HTTP requests to your Flask API.

Required Files for Render

💡 Why Gunicorn instead of Flask's built-in server?

Flask's development server (python app.py) only handles one request at a time and isn't safe for production. Gunicorn (Green Unicorn) is a production-grade WSGI server that runs multiple worker processes simultaneously, making it much faster and more reliable under real traffic.

How Render Builds Your App

1

Render clones your GitHub repo

It downloads the latest code from github.com/metadore/watersortgame

2

Render runs the build command

pip install -r requirements.txt — installs Flask, flask-cors, and gunicorn

3

Render runs the start command

gunicorn app:app — starts the Flask app with Gunicorn. The app:app means "from the file app.py, use the Flask object named app"

4

Your API is live!

Render assigns a URL like https://watersortgame-backend.onrender.com. Any HTTP request to this URL reaches your Flask server.

⚠️ Free Tier Limitation

On Render's free plan, the server spins down after 15 minutes of no traffic. The next request causes a cold start (~30 seconds delay). This is normal. Paid plans stay always-on.

A Complete User Journey

1

User opens the game URL

Browser requests https://watersortgame.vercel.app — Vercel CDN returns index.html in milliseconds from the nearest data center

2

Browser loads CSS and JS

Browser reads the HTML, sees links to styles.css, app.js, etc. and fetches them all from Vercel CDN

3

JavaScript initializes the game

app.js runs — calls GET https://watersortgame-backend.onrender.com/api/start-level?level=1. Render wakes up (if sleeping), Flask processes the request, returns tube data as JSON.

4

Tubes render on screen

JavaScript receives the JSON, calls renderTubes(data.tubes) — creates HTML elements dynamically, applies CSS gradient classes. Player sees the game!

5

Player clicks tubes

Each click triggers onTubeClick() → executePour() → POST /api/move to Render → Flask validates → JSON back → DOM updates. The cycle repeats for every move.

Glossary — Key Terms

Simple definitions for every technical word used in this project.

TermSimple Definition
APIA way for two programs to talk to each other. Your frontend "asks" the backend for data, and the backend "answers" with JSON.
REST APIA specific style of API that uses HTTP methods (GET, POST) and URLs to define what action to take.
JSONJavaScript Object Notation — a text format for sending data between programs. Looks like: {"key": "value", "num": 42}
HTTPThe protocol (set of rules) that browsers and servers use to communicate. GET means "give me data", POST means "here's data, process it".
DOMDocument Object Model — the live representation of your HTML page in the browser. JavaScript can read and modify it to change what you see.
CDNContent Delivery Network — a network of servers worldwide that store copies of your files so users download from the closest one.
CORSA browser security feature that blocks requests to different domains. Flask-CORS adds headers to allow your Vercel frontend to call your Render backend.
SessionA unique identifier (UUID) for one player's game. Used to look up their game state on the server.
BFSBreadth-First Search — explores all possible moves level by level, guaranteeing the shortest solution.
A*A search algorithm that uses a heuristic (educated guess) to explore the most promising paths first — faster than BFS for complex puzzles.
HeuristicAn estimate or "rule of thumb" that guides a search algorithm. In A*, it estimates how close a state is to the solution.
SQLiteA simple file-based database. All data lives in a single .db file. No separate database server needed.
GunicornA production Python web server. More powerful than Flask's built-in server — handles multiple requests at once.
GitA tool that tracks changes to your code files, letting you undo changes and see history.
CommitA saved snapshot of your code at a specific point in time, with a description of what changed.
Repository (Repo)A folder containing your code plus its entire Git history.
DeployMaking your application available on the internet by uploading it to a cloud server.
Environment VariableA configuration value (like an API key or URL) stored outside your code so it can be different in production vs development.
Async/AwaitJavaScript syntax for handling operations that take time (like API calls) without freezing the browser.
DecoratorIn Python, @app.route() is a decorator — it "decorates" a function with extra behaviour (making it handle a URL).
ClassA blueprint for creating objects. Tube and GameEngine are classes — each game has its own instances of these.
localStorageA browser storage area that persists data even after the page is closed. Used for dark mode preference and session ID.
fetch()A built-in JavaScript function for making HTTP requests to APIs. Returns a Promise (async result).
RegexRegular Expression — a pattern for matching text. Used to parse voice commands like "pour red into tube 2".
