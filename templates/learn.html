Step 01

Tube Click Handler

When a player clicks a tube, the browser captures the click event and routes it through a two-phase interaction system: source selection, then destination selection.

1
First Click — Select Source
If no tube is selected, the clicked tube becomes the source. It is visually highlighted with a cyan glow border.
2
Same Tube — Deselect
Clicking the same tube again cancels the selection gracefully.
3
Second Click — Pour Attempt
A different tube is clicked → the frontend sends a POST /api/move request with from_tube and to_tube.
Frontend — app.jsJavaScript
async function onTubeClick(idx) {
  if (State.selectedTube === null) {
    State.selectedTube = idx;           // Phase 1: select source
    setTubeSelected(idx, true);
    return;
  }
  const src = State.selectedTube;
  setTubeSelected(src, false);
  State.selectedTube = null;
  await executePour(src, idx); // Phase 2: pour
}
UX note: The selected tube state is held in-memory (never sent to backend). The backend is stateless per request — session state lives in the Python sessions dict.

@app.route("/api/move", methods=["POST"])
def make_move():
    data = request.get_json()
    engine = sessions[data["session_id"]]["engine"]
    success, reason, moved = engine.pour(
        data["from_tube"], data["to_tube"]
    )
    return jsonify({
        "success": success,
        "tubes": engine.snapshot(),
        "is_won": engine.is_won(),
        "move_count": engine.move_count,
    })
Step 03

Move Validation

All validation happens inside GameEngine.can_pour(src, dst). The rules are applied in strict priority order:

✓
Same-tube check
src == dst → rejected immediately.
✓
Source not empty
Can't pour from an empty tube.
✓
Destination not full
Tube at capacity → no room to receive.
✓
Color match (or empty destination)
Top color of source must equal top color of destination, OR destination is empty.
Backend — game_engine.pyPython
def can_pour(self, src: int, dst: int):
    s, d = self.tubes[src], self.tubes[dst]
    if s.is_empty():   return False, "Empty source"
    if d.is_full():    return False, "Destination full"
    if d.is_empty():   return True,  "OK"
    if s.top_color() != d.top_color():
        return False, "Color mismatch"
    return True, "OK"
Key insight: When pouring, all consecutive top-matching layers move together. If T1 has [red, red, blue] and T2 is empty, pouring moves two red layers at once.
Step 04

State Management

The game state is a pure Python dict, serializable to JSON at any time. The undo stack stores deep-copy snapshots before each move.

State SchemaJSON
{
  "tubes": [
    { "capacity": 4, "layers": ["red", "blue", "red"] },
    { "capacity": 4, "layers": [] }
  ],
  "move_count": 7,
  "click_heatmap": { "0": 3, "1": 8 },
  "move_history": [[0,1],[2,0]]
}

The state is persisted to SQLite via models.save_game() after each successful move, enabling save/load across browser sessions.

Step 05

UI Re-render

After a successful API response, the frontend performs a targeted partial re-render — only updating the liquid layers inside each tube, not rebuilding the entire DOM.

Frontend — app.jsJavaScript
function updateTubeState(tubes) {
  const wraps = document.querySelectorAll(".tube-wrap");
  tubes.forEach((tube, idx) => {
    const tubeEl = wraps[idx].querySelector(".tube");
    tubeEl.innerHTML = "";                    // clear layers
    tube.layers.forEach((color) => {
      const layer = document.createElement("div");
      layer.className = `liquid-layer layer-${color}`;
      layer.style.height = `${layerH}px`;
      tubeEl.appendChild(layer);
    });
  });
}
Gradient layers: Each color class (e.g. .layer-red) applies a CSS linear-gradient background, giving the multicolor liquid appearance. The shimmer animation is handled by a ::before pseudo-element.
Step 06

AI Solver (BFS / A*)

The AI solver uses BFS for small puzzles (guaranteed optimal) and falls back to A* for complex states.

Current State
→
Generate Moves
→
Apply + Hash
→
Visited Set
→
Win Detected
Backend — ai_solver.pyPython
def _bfs(self, start):
    queue = deque([start])
    visited = {start.key()}
    while queue:
        state = queue.popleft()
        for src, dst in state.legal_moves():
            nxt = state.apply_move(src, dst)
            if nxt.key() in visited: continue
            if nxt.is_won(): return nxt.path
            visited.add(nxt.key())
            queue.append(nxt)
    return None

The heuristic for A* counts the number of tubes not yet complete. Each non-finished tube adds 1 to the cost estimate — guiding the search toward solutions faster.

Step 07

Dynamic Difficulty

After each level is solved, the game compares the player's time to the expected time and adjusts a continuous difficulty_score (0.0–1.0).

Backend — difficulty_manager.pyPython
def adjust(self, level, time_taken, expected_time):
    ratio = time_taken / max(expected_time, 1)
    if   ratio < 0.5:  delta = +0.15   # very fast
    elif ratio < 0.8:  delta = +0.07
    elif ratio < 1.2:  delta = 0.0
    elif ratio < 1.8:  delta = -0.07
    else:              delta = -0.15   # very slow
    self.difficulty_score = max(0.0, min(1.0,
        self.difficulty_score + delta))

The difficulty score influences procedural level generation: tube count, color count, number of empty tubes, and scramble depth all scale with it.
